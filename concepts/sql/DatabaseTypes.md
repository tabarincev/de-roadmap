# Реляционные
## Модель данных
**`Реляционную модель`** предложил британский математик [Эдгар Кодд](https://ru.wikipedia.org/wiki/Кодд,_Эдгар) на рубеже шестидесятых и семидесятых годов XX в. Она основывается на теории множеств и представляет все данные в привычной, строгой и логичной форме обыкновенных таблиц. Такие таблицы в математике именуют [отношением](). Кодд сформулировал единые для любой реляционной базы данных законы и заложил основы построения запросов для работы с данными.

Данные в реляционной базе данных формируют отношения — двумерные таблицы с информацией о сущностях, т. е. объектах. Строка такой таблицы называется кортежем. Кортежи содержат множество атрибутов одной сущности, категории которых задаются в столбцах.

## Характеристики
| Признак                | Описание         |
|:-----------------------|:------------------|
| **Множество сущностей**    | Объекты со строго определенным набором атрибутов, с помощью которых они связываются между собой, формируют понятную и простую для восприятия структуру.             | 
| **Табличный формат**   | Такой формат гарантирует высокий уровень структурированности с жесткими логическими взаимосвязями, минимальный уровень избыточности данных, их согласованность и целостность.            | 
| **Язык SQL**    | SQL является стандартизированным средством общения пользователя с базой данных. Он очень формальный, что делает его удобным и простым в изучении. SQL гарантирует точный результат даже при сложном многоуровневом запросе.          | 
| **Масштабирование по вертикали**   | Реляционные базы данных хорошо масштабируются по вертикали. Но это значит, что по мере накопления информации в какой-то момент ее обработка потребует больших аппаратных ресурсов и финансовых затрат.          |
| **Масштабирование по горизонтали** | Горизонтальное масштабирование, подразумевающее распределение таблиц данных по множеству серверов, является слабой стороной реляционных баз данных. С разрастанием системы появляются задержки в обновлении данных. В какие-то моменты нарушается принцип целостности данных, что может негативно отразиться на пользовательском опыте. |
| **Наличие требований к параметрам данных** | Реляционные базы данных умеют работать только со структурированными данными. Но современный цифровой мир полон неструктурированных данных (например, фото и видео), к которым нельзя применять принципы реляционной модели. |

## Термины

| Термин | Определение | Пример | Особенности |
|:-------|:------------|:-------|:------------|
| **Таблица** | Основная структура для хранения данных | Таблица `Students` хранит информацию о студентах | |
| **Строка** | Представляет собой отдельную запись (экземпляр) в отношении | Одна строка в таблице `Students` представляет данные об одном студенте | |
| **Столбец** | Соответствует атрибуту отношения, описывает определенное свойство данных | В таблице `Students` могут быть столбцы - `name`, `age`, `faculty` | |
| **Схема** | Определяет ее структуру: имена столбцов, их типы данных и ограничения | В таблице `Students` - `{"name": "text", "age": "integer", "faculty": "text"}` | |
| **Первичный ключ** | Атрибут или набор атрибутов, который однозначно идентифицирует строку в таблице | В таблице `Students` первичным ключом может быть `student_id` | <ul><li>Значение `первичного ключа` уникально (`UNIQUE`) для каждой строки</li><li>Первичный ключ не может содержать `NULL`. </li></ul> |  
| **Внешний ключ** | Атрибут или набор атрибутов, который ссылается на первичный ключ другой таблицы, чтобы установить связь между таблицами | В таблице `Courses` внешний ключ `student_id` может ссылаться на `student_id` в таблице `Students` | `Внешний ключ` поддерживает целостность данных между связанными таблицами |
| **Отношение** | Представляет собой множество строк ([Tuples]()). [Таблица]() в реляционной базе данных — это реализация отношения | | |
| **Кортеж** | Набор значений, соответствующих всем атрибутам строки в таблице | ```{"ID студента": 1, "Имя": "Иван", "Возраст": 20, "Факультет": "Физика"}``` | |

# Нереляционные
## 1. Документо-ориентированные
`Документно-ориентированные базы данных` — это тип баз данных, которые хранят данные в виде документов, обычно в формате `JSON`, `BSON`, `XML` или других похожих структурах. Они предназначены для работы с неструктурированными или полуструктурированными данными.

### Особенности
- **`Гибкость схемы данных`**
  - Нет строгой схемы, как в реляционных базах данных.
  - Структура документов может изменяться динамически.
- **`Хранение иерархических данных`**
  - Легко хранить вложенные объекты.
  - Нет необходимости использовать сложные JOIN-запросы.
- **`Масштабируемость`**
  - Хорошо подходят для горизонтального масштабирования.
- **`Оптимизация под чтение`**
  - Быстрый доступ к документам по ключам.

### Термины
- **`Документ`**
  - Основная единица хранения данных, содержит данные в формате `JSON` или `BSON`.
- **`Коллекция`**
  - Группа документов, аналог таблицы в реляционных базах.
- **`Ключ`**
  - Уникальный идентификатор документа.
- **`Индекс`**
  - Используется для ускорения поиска документов.
- **`Запрос`**
  - Поиск или фильтрация документов по критериям
  
## 2.Графовые
`Графовые базы данных` — это тип NoSQL-баз данных, которые используют графовую модель для хранения данных. Они особенно эффективны для работы с сильно связанными данными, такими как социальные сети, рекомендации или дорожные карты.

### Особенности
- **`Гибкость модели`**
  - Легко добавлять новые узлы и связи без изменения структуры.
- **`Высокая производительность`**
  - Быстрые запросы для сильно связанных данных.
- **`Удобство работы с графами`**
  - Прямое отображение реальных взаимосвязей
- **`Идеально подходят для задач, связанных с графами`**
  - Социальные сети, рекомендации, дорожные карты.

### Термины
- **`Граф (Graph)`**
  - Состоит из `узлов` и `рёбер`.
- **`Узел (Node)`**
  - Сущность в `графе`.
- **`Ребро (Edge)`**
  - Связь между двумя `узлами`.
- **`Метки (Labels)`**
  - Определяют тип `узла` или `рёбра` (например, `Person`, `Company`).
- **`Свойства (Properties)`**
  - Дополнительные данные для `узлов` или `рёбер`.
  
## 3. Ключ-значение
`Базы данных ключ-значение` — это простой и высокопроизводительный тип NoSQL-баз данных, в которых данные хранятся в виде пар `ключ-значение`. Они идеально подходят для задач, требующих быстрой записи и чтения.

### Особенности
- **`Простота структуры`**
  - Нет сложных схем данных или таблиц.
- **`Высокая производительность`**
  - Быстрая запись и чтение данных.
- **`Горизонтальное масштабирование`**
  - Легко распределять данные между серверами.
- **`Гибкость`**
  - Значения могут быть любого типа: от строк до сложных объектов.
- **`Отсутствие поддержки сложных запросов`**
  - Не подходит для анализа данных или сложных выборок.

### Термины
- **`Ключ`**
  - Уникальный идентификатор записи.
- **`Значение`**
  - Любой тип данных, связанный с ключом (строка, число, JSON, двоичные данные и т.д.).

## 4. Колоночные
`Колоночные базы данных` — это тип NoSQL баз данных, где данные хранятся в виде столбцов вместо традиционного хранения строк. Этот подход оптимален для аналитических задач и работы с большими объемами данных.


## Сравнение
# Сравнение типов NoSQL баз данных

| **Критерий**               | **Документные** | **Ключ-значение** | **Колоночные** | **Графовые**         |
|-----------------------------|----------------------------|-------------------------------|----------------------------|------------------------------|
| **Примеры**                | [MongoDB](https://www.mongodb.com), [CouchDB](https://couchdb.apache.org), [Firebase](https://firebase.google.com/docs/database?hl=ru)| Redis, Memcached, DynamoDB    | Cassandra, HBase, ClickHouse | Neo4j, ArangoDB, Amazon Neptune |
| **Модель данных**          | Документы (JSON, BSON)    | Пары `ключ-значение`         | Семейства столбцов         | Узлы, рёбра, свойства        |
| **Структура хранения**     | Гибкая (без строгой схемы)| Простая (ключ и значение)    | Столбцы, сгруппированные в семейства | Графы: узлы и связи          |
| **Основные операции**      | CRUD по документам        | CRUD по ключу                | Чтение/запись столбцов     | Поиск и анализ связей       |
| **Скорость чтения**        | Быстрая                   | Очень быстрая                | Высокая для отдельных столбцов | Оптимизирована для графовых запросов |
| **Скорость записи**        | Быстрая                   | Очень быстрая                | Высокая                    | Зависит от размера графа     |
| **Масштабируемость**       | Горизонтальная            | Горизонтальная               | Горизонтальная             | Горизонтальная, но сложнее   |
| **Подходит для**           | Неструктурированных данных (IoT, блоги) | Кэширование, управление сессиями | Аналитики, хранилищ данных | Социальных сетей, рекомендаций |
| **Сложность запросов**     | Средняя (поддержка сложных фильтров) | Простая                     | Средняя (оптимизировано для агрегаций) | Сложная (Cypher, Gremlin)    |
| **Поддержка транзакций**   | Ограниченная              | Минимальная                  | Ограниченная               | Хорошая                     |
| **Преимущества**           | Гибкость схемы, вложенные данные | Высокая скорость, простота  | Эффективность в аналитике  | Моделирование связей        |
| **Недостатки**             | Не подходит для сложных связей | Не поддерживает сложные операции | Сложность настройки и администрирования | Может быть медленнее на больших графах |
| **Сценарии использования** | Управление содержимым, аналитика | Кэширование, управление конфигурацией | Большие данные, временные ряды | Социальные сети, маршруты    |
